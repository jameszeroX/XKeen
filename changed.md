# Security Hardening — Усиление безопасности XKeen

## Описание

Комплексное усиление информационной безопасности XKeen: верификация целостности загружаемых бинарников, защита конфиденциальных данных, устранение небезопасных паттернов кода и улучшение управления процессами.

## Что исправлено

### 1. Верификация целостности загружаемых бинарников

**Проблема:** Xray и Mihomo скачивались без какой-либо проверки подлинности. При использовании сторонних GitHub-прокси (`ghfast.top`, `gh-proxy.com`) атакующий мог подменить бинарник — и он устанавливался бы с правами на исполнение в `/opt/sbin/`.

**Решение:**

- Создан модуль `08_tools_integrity.sh` с функциями:
  - `verify_sha256()` — вычисляет SHA256 файла и сравнивает с ожидаемым (поддержка `sha256sum` и `openssl`)
  - `fetch_xray_dgst_sha256()` — загружает и парсит `.dgst` файл из релизов Xray
  - `verify_download_integrity()` — обёртка с выводом результата пользователю
- **Xray:** после загрузки скачивается `.dgst` файл **напрямую с GitHub** (не через прокси) и сверяется SHA2-256. При несовпадении файл удаляется, установка прерывается
- **Mihomo:** SHA256 digest извлекается из метаданных GitHub API release assets (поле `digest`). Запрос к API идёт **напрямую**, минуя прокси-зеркала
- При отсутствии `sha256sum`/`openssl` на устройстве — выводится предупреждение, загрузка не блокируется (soft fail)

### 2. Устранение генерации shell-скриптов через heredoc и санитизация

**Проблема:**
- Функции `restore_backup_xkeen()` и `install_xkeen()` создавали скрипты через `mktemp` и `heredoc` с немедленным `exec`, что могло привести к инъекции кода.
- Функция `configure_firewall()` генерировала массивный ~500-строчный bash-скрипт `proxy.sh`, внедряя в него более 30 переменных напрямую через `heredoc` (вида `name_client="$name_client"`). Если любая из переменных содержала спецсимволы (двойные кавычки, доллар и т.д.), это ломало скрипт или создавало уязвимость. Кроме того, в исходном коде были баги экранирования (например, случайное вычисление переменной `$_r` или `|$` на этапе генерации скрипта).

**Решение:**
- В `restore_backup_xkeen()` и `install_xkeen()` логика выполняется прямо в текущем процессе, генерация временных скриптов полностью удалена.
- В `configure_firewall()` внедрена строгая санитизация:
  1. Блок генерации разделен. Теперь переменные безопасно эскейпятся и записываются в одинарных кавычках через специальную функцию `inject_var`.
  2. Тело скрипта записывается через `cat <<'EOL'` (с кавычками), что полностью отключает интерпретацию переменных внутри heredoc.
  3. Исправлены баги со случайно вычисляемыми переменными вроде `$_r`.

### 3. Ужесточение прав доступа на netfilter hook

**Проблема:** Генерируемый скрипт `proxy.sh` (управление iptables/ip6tables) создавался с `chmod +x`, что делало его читаемым для всех пользователей. Файл содержит метки политик, порты прокси, IP-адреса исключений.

**Решение:** Права изменены на `chmod 700` — только владелец (root) может читать, изменять и исполнять.

### 4. Graceful shutdown прокси-клиента

**Проблема:** `proxy_stop()` использовала `killall -q -9` (SIGKILL) без предварительной попытки корректного завершения. SIGKILL не позволяет процессу закрыть соединения, сохранить состояние и освободить ресурсы.

**Решение:** Реализована двухэтапная остановка:
1. `SIGTERM` — даёт процессу 3 секунды на корректное завершение
2. `SIGKILL` — только если процесс не завершился после SIGTERM

### 5. Безопасность временных файлов (mktemp)

**Проблема:** Временные файлы (для скачивания обновлений, геофайлов, конфигураций) создавались в системной директории `/tmp` с помощью вызова `mktemp`. На роутерах каталог `/tmp` располагается в оперативной памяти (RAM). Загрузка больших архивов могла привести к исчерпанию RAM и перезагрузке устройства (OOM), а также к конфликтам имён.

**Решение:** Все вызовы `mktemp` переведены в локальные рабочие директории XKeen (например, `/opt/tmp/xkeen`, `/opt/tmp/xray`, `/opt/tmp/mihomo`). Теперь временные файлы хранятся на внешнем носителе, что исключает OOM, и имеют префиксы, что облегчает их идентификацию.

### 6. Защита от параллельного запуска (Race Conditions)

**Проблема:** Отсутствовал механизм блокировки (lockfile) при выполнении операций установки, обновления ядер (XKeen, Xray, Mihomo) или изменения конфигурации. Запуск нескольких экземпляров `xkeen` одновременно (например, вручную и по cron) мог привести к повреждению файлов и состояния пакета.

**Решение:** 
- В главный скрипт `xkeen` внедрена функция `acquire_lock`.
- При вызове команд, модифицирующих систему (`-install`, `-uk`, `-ux`, `-um`, `-ug`, `-dk` и др.), создается лок-файл `/tmp/xkeen_action.lock`, в который записывается PID процесса.
- Если лок-файл существует, проверяется активность процесса (`kill -0`). В случае обнаружения активного конкурентного процесса операция прерывается с ошибкой.
- При завершении работы (в том числе по сигналам `INT`, `TERM`) лок-файл гарантированно удаляется через `trap`. Реализована поддержка безопасного перезапуска скрипта через `exec` (проверка `lock_pid != $$`).

### 7. Защита конфиденциальных данных в диагностике

**Проблема:** Команда `xkeen -diag` записывала в `/opt/diagnostic.txt` полное содержимое конфигурационных файлов Xray (dns, inbounds, routing), включая UUID серверов, пароли, приватные ключи и IP-адреса. Файл создавался без ограничения прав доступа.

**Решение:**
- Файл `diagnostic.txt` создаётся с `chmod 600` (только владелец)
- Добавлена функция `mask_sensitive_data()`, которая маскирует в JSON-конфигах:
  - `id` (UUID) → `***MASKED***`
  - `password` → `***MASKED***`
  - `privateKey`, `secretKey`, `shortId` → `***MASKED***`
  - `key` (длиннее 16 символов) → `***MASKED***`
  - `address` (IPv4) → `***IP_MASKED***`
- Конфиги записываются через `mask_sensitive_data` вместо прямого `log_file`
- Пользователю выводится предупреждение о содержимом файла и информация о маскировке

### 8. Защита от Path Traversal в обработчике геофайлов

**Проблема:** В функции `process_geo_file()` имя файла сохранялось напрямую без валидации, что могло позволить атакующему (в случае внедрения пути через переменные) перезаписать произвольные файлы в системе, используя символы `../` или абсолютные пути `/`.

**Решение:**
- Добавлена строгая проверка параметра `filename` в начале функции. Имена файлов, содержащие `/`, `\`, `..` или `.` отвергаются с ошибкой.

### 9. Удаление `opkg upgrade` из `test_entware()`

**Проблема:** Функция `test_entware()` при каждом запуске установки выполняла `opkg upgrade >/dev/null 2>&1`, что молча обновляло **все** установленные пакеты Entware (OpenSSL, bash, wget, busybox и др.) без ведома и согласия пользователя. Это создавало несколько рисков:

1. **Нарушение принципа наименьших привилегий** — пользователь ожидает установку XKeen, а не полное обновление системы.
2. **Риск поломки системы** — обновление может сломать зависимости других пакетов, изменить ABI shared-библиотек или переполнить хранилище на устройствах с ограниченным flash.
3. **Supply-chain риск** — `opkg upgrade` загружает и устанавливает произвольный код из репозитория Entware. При компрометации зеркала атакующий получает выполнение кода на роутере без предупреждений.
4. **Нарушение идемпотентности** — повторный запуск `xkeen -install` давал разный результат в зависимости от наличия обновлений в репозитории.

**Решение:** Вызов `opkg upgrade` удален. Оставлен `opkg update` (безопасное обновление индекса пакетов) и `install_packages` (установка только конкретных зависимостей XKeen).

### 10. SHA256 верификация для Yq и XKeen

**Проблема:** Yq и XKeen также скачивались без проверки целостности.

**Решение:**
- В GitHub Actions для сборки XKeen (`package-folder.yml`) добавлена генерация файла `xkeen.tar.gz.sha256`.
- В `install.sh` и `02_donwloaders_xkeen.sh` добавлена загрузка `.sha256` файла и проверка `xkeen.tar.gz`.
- В `01_downloaders_mihomo.sh` добавлена проверка Yq через извлечение `digest` (SHA256) из GitHub API (как и для Mihomo).

## Флаг `-autoinstall` — Автоматическая установка без интерактивных запросов

### Описание

Добавлен новый флаг `xkeen -autoinstall`, который выполняет полный цикл установки без каких-либо диалогов с пользователем. Предназначен для автоматизированного развёртывания (скрипты, CI, массовая установка).

**Параметры по умолчанию:**
- Ядро проксирования: **Xray** (последняя стабильная версия)
- Геофайлы: **все шесть баз** — Re:filter, V2Fly, ZKeen (GeoSite) + Re:filter, V2Fly, ZKeenIP (GeoIP)
- Расписание автообновления: **суббота 03:00** (`0 3 * * 6 xkeen -ug`)
- Автозапуск при старте роутера: **включён** (по умолчанию)

### Изменённые файлы

- `scripts/xkeen` — добавлен блок `-autoinstall)` в основной цикл обработки аргументов
- `scripts/_xkeen/04_tools/07_tools_downloaders/01_downloaders_xray.sh` — добавлена ветка авто-режима в функции `download_xray()`
- `scripts/_xkeen/about.sh` — добавлена строка справки для флага

### Технические детали

**`scripts/xkeen`:**
Новый блок повторяет последовательность вызовов флага `-i`, но вместо интерактивных функций `choice_*` напрямую устанавливает переменные:
- `add_xray=true`, `add_mihomo=false`
- `install_refilter_geosite`, `install_v2fly_geosite`, `install_zkeen_geosite` = `true`, `bypass_cron_geosite=false`
- `install_refilter_geoip`, `install_v2fly_geoip`, `install_zkeenip_geoip` = `true`, `bypass_cron_geoip=false`
- `choice_geofile_cron_select=true`, `choice_geofile_cron_time="0 3 * * 6"`
- Вызов `choice_autostart_xkeen` пропускается — автозапуск включён по умолчанию

Перед вызовом `download_xray` устанавливается `autoinstall_mode=true`, после — сбрасывается через `unset`.

**`01_downloaders_xray.sh`:**
После получения списка релизов добавлен блок проверки `$autoinstall_mode`. При активном режиме:
1. Берётся первый (новейший) тег из `$RELEASE_TAGS` — `head -1`
2. Добавляется префикс `v` при использовании jsDelivr
3. Выводится сообщение о выбранной версии
4. Выполняется загрузка с полной верификацией SHA256
5. При любой ошибке — `exit 1` (нет повторного меню)

### Использование

```sh
xkeen -autoinstall          # стандартный запуск
xkeen -autoinstall -toff    # с отключённым таймаутом (медленный GitHub)
```
